第5回 ソフトウェアと言語処理系

講義

大きな絵

前回までで「ハードウェアは CPU + メモリ で、機械語を実行する」ことを見た。今回は ソフトウェア側。人間が書いたプログラムが、どのようにして CPU が実行できる形（機械語）に変換され、動くのか。鍵になるのは 抽象化 と 翻訳（コンパイル）／逐次実行（インタプリト） の 2 つの戦略。

1. 計算機ソフトウェアとは

計算機システムは ハードウェア（変更しにくい機械の部分）と ソフトウェア（命令の集まり）から成る。

フォン・ノイマンの核心アイディア。「プログラムもデータと同じく メモリに格納する」。これによってメモリの中身を書き換えるだけで、同じハードウェアがまったく違う作業をこなせるようになる。

2. アルゴリズムとプログラムの違い

  • アルゴリズム：計算の手順を厳密・曖昧さなく記述したもの。計算機とは独立。
  • プログラム：アルゴリズムを特定の計算機・言語で書き直したもの。

カーニハン流に言えば「アルゴリズム＝設計図、プログラム＝建物」。同じアルゴリズムから、C / Python / アセンブリで異なるプログラムが作れる。

例：ユークリッドの互除法（紀元前 3 世紀、世界最古のアルゴリズム）

2 つの自然数 $n, m$ の最大公約数 GCD を求める：

  1. $k \leftarrow n \bmod m$
  2. $k = 0$ なら $m$ が GCD（終了）
  3. そうでなければ $(n, m) \leftarrow (m, k)$ として 1. に戻る

3. アセンブリ言語と高水準プログラミング言語

機械語 → アセンブリ言語

初期は機械語（0/1 の列）を直接書いていた。あまりに人間に不向き。そこで機械語命令に ニモニック（ADD, BZ など）と メモリ番地への名前付け を導入したのが アセンブリ言語。アセンブリ → 機械語の変換は アセンブラ というプログラムが行う。

ソフトウェア技術の核心思想：「プログラムをデータとして、プログラムの加工をプログラムで行う」。アセンブラ自身もプログラム。後に出てくるコンパイラ・インタプリタもすべてこの考え方の発展形。

高水準プログラミング言語の登場

アセンブリ言語の欠点：

  • 命令が単純すぎて、アルゴリズムとの 抽象度の乖離 が大きい
  • CPU が変わると命令体系も変わる → プログラム全書き直し

これを解決するのが 高水準言語。最初の高水準言語は FORTRAN（FORmula TRANslation system）。(x + y) * 2 のような数式の自然な書き方がそのまま使える。

例：C 言語でのユークリッド互除法

#include <stdio.h>

int main(void) {
    int n, m, k;
    scanf("%d, %d", &n, &m);
    printf("GCD of %d and %d is ", n, m);
    while ((k = n % m) != 0) {
        n = m;
        m = k;
    }
    printf("%d\n", m);
    return 0;
}

抽象化（abstraction）

高水準言語ではレジスタの数や命令セットといった「計算機の詳細を意識しないで」プログラムが書ける。このように 詳細を忘れてよい仕組みを提供する ことを 抽象化 と呼ぶ。計算機科学全体で最も重要な考え方の一つ。

構文と意味

  • 構文（syntax）：どんな記号列がプログラムとして認められるか（文法）
  • 意味（semantics）：プログラムを実行するとどんな結果になるか（動作）

4. コンパイラとインタプリタ

高水準言語のプログラムはそのままでは CPU で動かない。2 つの戦略がある。

料理にたとえると。
  • コンパイラ：日本語レシピを丸ごと英語に翻訳して料理人に渡す
  • インタプリタ：日本語レシピを横で読み上げながら料理人が逐次調理する

言語処理系の「3 つの言語」

コンパイラ・インタプリタなど「高水準言語プログラムを処理するソフトウェア」を総称して 言語処理系 という。1 つの言語処理系には必ず 3 種類の言語が関わる：

用語意味例（GCC の場合）
ソース言語処理系が入力として受け取る言語C 言語
ターゲット言語コンパイラが出力する言語x86 機械語
実装言語言語処理系自身が書かれている言語C 言語（GCC 自体も C で書かれている）
身近な例で整理
  • Python (CPython)：ソース＝Python、実装＝C
  • TypeScript コンパイラ tsc：ソース＝TS、ターゲット＝JS、実装＝TS
  • Java コンパイラ javac：ソース＝Java、ターゲット＝JVM バイトコード、実装＝Java

ソース言語と実装言語が一致するのが セルフホスティングコンパイラ（後述）。

よくある誤解：「コンパイラ言語／インタプリタ言語」は正確ではない。どんな言語も原理的にはコンパイラでもインタプリタでも実装できる。「Python は遅い」というのも、CPython の実装上の話で、PyPy のような JIT コンパイラ実装もある。

4.1 コンパイルの実例

C の while ((k = n % m) != 0) { n = m; m = k; } を GCC で x86 アセンブリにコンパイルすると：

    jmp .L2
.L3:
    movl  -16(%rbp), %eax     ; n = m
    movl  %eax,  -20(%rbp)
    movl  -12(%rbp), %eax     ; m = k
    movl  %eax,  -16(%rbp)
.L2:
    movl  -20(%rbp), %eax     ; n の値をレジスタへ
    movl  -16(%rbp), %ecx
    cltd
    idivl %ecx                ; eax ÷ ecx → 商 eax, 余り edx
    movl  %edx, -12(%rbp)     ; 余りを k に格納
    cmpl  $0, -12(%rbp)       ; k と 0 を比較
    jne   .L3                 ; k ≠ 0 なら .L3 へ

C の 4 行 がアセンブリでは 16 行。高水準言語の 簡潔さ＝抽象化のメリットがはっきり見える。

4.2 コンパイラのバリエーション

種類説明
セルフホスティングソース言語と実装言語が同じコンパイラ（自分自身でビルドできる）
クロスコンパイラ実行環境と 違う CPU 向けの機械語を生成（例：PC 上で ARM 用のコードを作る）
source-to-source（トランスパイラ）ターゲットも高水準言語のコンパイラ（例：TypeScript → JavaScript）

4.3 発展形：JIT・仮想機械・最適化

エミュレータ・仮想機械（Virtual Machine）

計算機 X の機械語を入力とするインタプリタを計算機 Y 上で動かすと、Y の上で X が動いているように見える。実在のハードを模倣するのが エミュレータ、抽象的に設計された計算機なら 仮想機械（VM）。

JIT コンパイル（just-in-time）

インタプリタが内部にコンパイラを抱え、実行中に 必要な部分だけを機械語に変換して走らせる方式。通常の「実行前に全部変換」する方式は ahead-of-time。インタプリタの柔軟さとコンパイラの速さの折衷。

最適化（optimization）

同じ意味のコードを、より速い／小さい機械語に変換するコンパイラ内の処理。例：1 + 1 + x はコンパイル時に 2 + x にしておけば加算命令が 1 つ減る（定数畳み込み）。

講義のまとめ

  • ソフトウェアは命令の集まり。フォン・ノイマン式では プログラムもメモリに置く ことで柔軟性を獲得
  • アルゴリズムは設計図、プログラムはそれを特定の計算機・言語で実装したもの
  • アセンブリ言語は機械語に名前を付けたもの。アセンブラ が機械語に変換する
  • 高水準言語は 抽象化 で計算機の詳細を隠す。同じプログラムを別の CPU で動かせる
  • コンパイラ：丸ごと翻訳してから実行。インタプリタ：読みながら逐次実行
  • 「コンパイラ言語／インタプリタ言語」という分類は厳密には誤り。どんな言語もどちらでも実装可能
  • 発展形に JIT コンパイル、仮想機械（JVM など）、最適化 がある
